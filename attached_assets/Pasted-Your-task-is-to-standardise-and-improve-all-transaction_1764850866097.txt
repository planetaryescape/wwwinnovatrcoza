Your task is to standardise and improve all transactional emails for Innovatr and to implement a full set of reusable HTML templates using the copy provided below.

The goal is that every email sent by the system:

Looks clean, simple and professional (similar to a HubSpot style but without images).

Uses one consistent layout wrapper.

Uses the same button style and typography.

Uses the copy and subject lines below verbatim.

Supports both HTML and a clean text fallback.

Works with the existing email service and database driven user accounts.

1. General design and architecture

Use a single HTML layout wrapper function, for example:

function renderBaseEmail(options: {
  title: string;
  greeting?: string;
  bodyHtml: string;
  buttonLabel?: string;
  buttonUrl?: string;
  showButton?: boolean;
  footerNote?: string;
}): string


Layout requirements:

White background, dark text, centred container max width ~600px.

Use simple, web safe fonts like system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif.

Header section with the email title in bold.

Greeting line, e.g. Hi {{FirstName}}, with a fallback to Hi there,.

Body copy in left aligned paragraphs with good spacing.

Optional primary button in a single consistent style:

Background colour: use the existing Innovatr brand button colour already used on the website or current password email (do not introduce a new colour variable).

White text, slightly rounded corners.

Under the button, when relevant, include a plain URL fallback line, e.g.
If the button does not work, copy and paste this link in your browser: followed by the URL.

Footer (always the same):

This is an automated message from Innovatr.
If you have any questions, contact us at hello@innovatr.co.za.
© 2025 Innovatr. All rights reserved.


Implement a central email template renderer in server/emails/email-service.ts (or the existing email service file) that:

Accepts a template type and a data payload.

Returns subject, html, and text.

Continue using the existing email delivery mechanism (Resend or current provider).

Make sure the email service calls renderEmailTemplate and passes the correct type and data.

Use a single FRONTEND_URL or similar base url from environment. If it does not exist, create one and default to https://www.innovatr.co.za.

Ensure all existing email calls are updated to use these templates:

Password reset requests.

Any current brief submission or system notifications.

Any existing order or subscription related emails.

For every email, also generate a plain text version by stripping tags from the HTML or by composing a minimal text alternative that matches the content.

2. Dynamic fields

For all templates, support these dynamic values where relevant:

firstName (fallback to generic “there”).

resetLink

verifyLink

portalLink (usually ${FRONTEND_URL}/portal/login or ${FRONTEND_URL}/portal/research depending on context)

reportLink if you have a deep link to a specific study or report.

basicCredits, proCredits

invoiceNumber, amount, orderReference

Make sure the renderer handles missing optional fields gracefully.

3. Email templates and copy

Implement all the templates below with exact subjects and wording.
Use the base layout wrapper for each, providing the specific title, body copy, and optional button.

Where you see {{placeholder}}, keep this as a dynamic variable to be interpolated from the data payload.

4. Integration and testing

Make sure each place in the codebase that sends an email now calls renderEmailTemplate with the correct type and data.

Confirm that:

Password reset flow uses PASSWORD_RESET_REQUEST and then PASSWORD_RESET_SUCCESS after change.

Account creation uses ACCOUNT_CREATED (and EMAIL_VERIFICATION if verification is enabled).

Brief submission uses BRIEF_SUBMITTED.

Audience live and study complete flows use AUDIENCE_LIVE and STUDY_COMPLETE.

Credit updates and low credit events call CREDITS_ADDED and CREDITS_LOW.

Payment and invoice flows call PAYMENT_CONFIRMED and INVOICE_ISSUED.

Log sample outputs for at least one example of each template to the server console for review during development.

Ensure no sensitive data such as password or passwordHash is ever included in the email payload or template.

When you are done, the system should be using this unified design and copy for all transactional emails.
