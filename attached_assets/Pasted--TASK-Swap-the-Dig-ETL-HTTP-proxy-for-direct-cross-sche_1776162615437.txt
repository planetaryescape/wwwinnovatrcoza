# TASK: Swap the Dig ETL HTTP proxy for direct cross-schema SQL reads

## Who you are

You're the same senior full-stack engineer who completed the Dig ETL integration in the previous session by following the prompt at `https://innovatr-dig-etl.vercel.app/docs/11-replit-agent-prompt`. That work is now in the repo: `dig_api_keys` table, `server/integrations/dig-etl.ts`, 9 proxy routes under `/api/member/dig/*`, React Query hooks at `client/src/lib/dig-api.ts`, components under `client/src/components/dig/`, the new `/portal/reports/:id` page, and the `RichStudyCard` branch in `TestPage.tsx`.

Innovatr decided the Bearer-key + HTTP proxy approach is unnecessary friction. The dashboard and the ETL share the same Neon Postgres database. The dashboard already enforces tenant isolation via `req.user.companyId` from its session. Going through HTTP and Bearer auth to read a sibling schema in the same database is overkill.

Your job: rip out the proxy implementation, replace it with direct SQL reads against the `dig.*` schema. Same routes at `/api/member/dig/*`, same JSON response shapes, same client code. **Only the server-side route handlers change.**

## The new architecture in one diagram

```
Before (current):                           After (this task):
─────────────────                           ──────────────────
React → /api/member/dig/studies             React → /api/member/dig/studies
       ↓                                           ↓
Express requireAuth                         Express requireAuth
       ↓                                           ↓

       ↓                                                WHERE company_id = $1)

       ↓

```

The hop from "Express requireAuth" to "SELECT" goes from network + Bearer + remote service to a single local database query. Everything before requireAuth and after the SELECT is unchanged.

## Why this is safe

The same things that made the proxy approach safe make the direct approach safe:

- The dashboard's session enforces tenant isolation: every authenticated request has `req.user.companyId` set by `server/middleware.ts` from the session cookie hash → user lookup → company link
- Every new SQL query in this task includes `WHERE company_id = $1` with that value
- The `dig.*` schema's tables ALL carry a denormalised `company_id` column for exactly this reason
- The dashboard never queries `dig.*` without a `company_id` filter — the prompt's done criteria block enforces this

The Bearer key approach added one extra layer of "the ETL also checks". With direct reads, the dashboard is the sole enforcement point, but it's the same enforcement point that already gates every other `/api/member/*` route in the repo. If the existing tenant isolation is trustworthy, this is too.

### 3. `GET /api/member/dig/studies/:id/concepts` — concept aggregates

This one has post-processing — the SQL returns raw counts, the route handler computes percentages.

```sql
SELECT
  c.id,
  c.name,
  c.concept_type,
  c.brand,
  c.product_name,
  c.product_format,
  c.segments,
  COUNT(DISTINCT si.id)::int AS interest_sample,
  COUNT(DISTINCT si.id) FILTER (WHERE si.interested)::int AS interested_yes,
  (SELECT COUNT(*)::int FROM dig.screening_commitment sc
    WHERE sc.won_concept_id = c.id) AS wins,
  (SELECT COUNT(*)::int FROM dig.screening_commitment sc
    WHERE sc.lost_concept_id = c.id) AS losses
FROM dig.concepts c
LEFT JOIN dig.screening_interest si ON si.concept_id = c.id
WHERE c.dig_study_id = ${req.params.id}
  AND c.company_id = ${companyId}
GROUP BY c.id, c.name, c.concept_type, c.brand, c.product_name, c.product_format, c.segments
ORDER BY c.name ASC
```

Then in JS, transform each row:

```ts
const concepts = rows.map((r: any) => {
  const commitmentSample = r.wins + r.losses;

    brand: r.brand,
```

### 9. `POST /api/member/dig/search` — semantic verbatim search

This is the only endpoint that's not just SQL — it needs to embed the user's query text via OpenAI before running the cosine-distance lookup. Two paths:

**Recommended path: add OpenAI to the dashboard.**

1. Add `OPENAI_API_KEY` to Replit Secrets (the same key Innovatr already uses for the ETL — ask Alroy/Hannah if you don't have it)
2. `npm install openai` (this is the one new dependency the previous prompt forbade — it's allowed here because it replaces the deleted `server/integrations/dig-etl.ts` rather than adding to it)
3. Create `server/integrations/dig-search.ts`:

```ts
import OpenAI from "openai";
import { sql } from "drizzle-orm";
import { db } from "../db";

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

export interface SearchRequest {
  query: string;
  study_id?: string;
  concept_id?: string;
  limit?: number;
}

export interface SearchResult {
  content: string;
  distance: number;
  source_table: "evaluations" | "eval_heatmap_clicks";
  source_id: string;
  concept_id: string | null;
}

export async function digSearch(
  body: SearchRequest,
  companyId: string,
): Promise<{ results: SearchResult[] }> {
  if (!body.query || typeof body.query !== "string") {
    throw new Error("query is required");
  }
  const limit = Math.min(Math.max(body.limit ?? 10, 1), 100);

  // Embed the query
  const embeddingResponse = await openai.embeddings.create({
    model: "text-embedding-3-small",
    input: body.query,
  });
  const queryVec = embeddingResponse.data[0].embedding;
  const queryVecLiteral = `[${queryVec.join(",")}]`;

  // Search
  const studyFilter = body.study_id
    ? sql`AND dig_study_id = ${body.study_id}`
    : sql``;
  const conceptFilter = body.concept_id
    ? sql`AND concept_id = ${body.concept_id}`
    : sql``;

  const results = await db.execute(sql`
    SELECT
      content,
      (embedding <=> ${queryVecLiteral}::vector) AS distance,
      source_table,
      source_id,
      concept_id
    FROM dig.text_embeddings
    WHERE company_id = ${companyId}
      ${studyFilter}
      ${conceptFilter}
    ORDER BY embedding <=> ${queryVecLiteral}::vector
    LIMIT ${limit}
  `);

  return { results: results as SearchResult[] };
}
```

4. Wire the route:
