3. Deals model, API and permissions

We already have an empty Deals tab. Now implement:

3.1 Deal model

Create a Deal model in the storage layer with at least:

id

title

description

headlineOffer – short display text, e.g. “5 Test24 Basic credits for R40k”

originalPrice – number (e.g. 50000 for R50k)

discountedPrice – number (e.g. 40000 for R40k)

discountPercent – number (can be computed automatically from original vs discounted)

creditsIncluded – number (how many credits the deal bundles)

targetTierKeys – array of membership tiers: "STARTER" | "GROWTH" | "SCALE" (can be empty)

targetUserIds – array of specific user IDs (can be empty)

ownerCompanyId – optional; if present, indicates this deal belongs to a specific company (for future wholesaler scenarios)

createdByUserId

validFrom – date

validTo – date or null

isActive – boolean

createdAt

Rules:

A deal is visible to a member if:

isActive = true

validFrom <= today

validTo is null or >= today

AND (

their id is in targetUserIds, OR

their membershipTier is in targetTierKeys
)

3.2 Backend endpoints

Create these endpoints:

Admin

GET /api/admin/deals

Returns all deals, with sufficient fields to show in the admin table.

POST /api/admin/deals

Creates a new deal.

Only users with role = "ADMIN" or "DEAL_ADMIN" may create.

If role = "DEAL_ADMIN", automatically set ownerCompanyId to that user’s company, and restrict any targeting logic as described below.

PATCH /api/admin/deals/:id

Updates deal fields.

ADMIN can update any fields.

DEAL_ADMIN can only update deals where ownerCompanyId matches their own company; they may not change ownerCompanyId.

DELETE /api/admin/deals/:id

Soft delete: set isActive = false.

Member

GET /api/member/deals

Uses the visibility rules above to only return deals relevant to the current user.

All admin endpoints must verify the user is authenticated and has role = "ADMIN" or role = "DEAL_ADMIN" as appropriate.
