## What the Dig ETL is (context)

The Dig ETL is a separate Next.js app that ingests raw Upsiide CSV bundles for a research study and writes the parsed data into 17 tables in our shared Neon Postgres under the `dig.*` schema. You do **not** read those tables directly ã you go through its REST API at `https://innovatr-dig-etl.vercel.app/api/dig/*`.

**Auth model:**

- The ETL issues one Bearer token per company, format `dig_live_<random>`.
- The token is scoped to exactly one `company_id` ã it can't read any other tenant's data.
- Every request sends `Authorization: Bearer dig_live_<key>` and the ETL filters every query by the token's company.
- Tokens are SHA-256 hashed at rest on the ETL side. The raw token is shown once at mint time and never again.

**Relationship to your existing schema:**

- The Dig ETL writes to `dig.studies`, which has a nullable foreign key `public_client_report_id` pointing at your existing `public.client_reports.id`. That's how you join the new data to a study the admin already knows about.
- When Hannah uploads Upsiide CSVs to the ETL's admin UI at `https://innovatr-dig-etl.vercel.app/admin/ingest`, she picks the client report ã that sets the FK for you.
- Your `public.companies` table is the same table the ETL uses. Same tenant boundary, just now with much richer data.

## Architecture decision ã already made, do not re-debate

**Server-side proxy.** All Dig ETL calls go through new Express routes on your backend at `/api/member/dig/*` which forward to `https://innovatr-dig-etl.vercel.app/api/dig/*`.

Why this, not the alternatives:

- ã **Direct client fetch from React to the ETL URL** ã would expose the Bearer key in the browser. Also requires CORS on the ETL side (not currently configured).
- ã **Read `dig.*` tables directly with Drizzle** ã technically possible since it's the same Neon, but it couples your deploys to their schema migrations, bypasses the ETL's tenant-isolation test gate, and defeats the "API over the wire" contract. Do not do this.
- ã **Server proxy** ã the key stays server-side, the call chain integrates with your existing `requireAuth` middleware, and you get one audit point for rate-limiting later.

```ts
import { pgTable, varchar, text, timestamp } from "drizzle-orm/pg-core";

export const digApiKeys = pgTable("dig_api_keys", {
  companyId: varchar("company_id")
    .primaryKey()
    .references(() => companies.id, { onDelete: "cascade" }),
  keyValue: text("key_value").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
  rotatedAt: timestamp("rotated_at"),
});

export type DigApiKey = typeof digApiKeys.$inferSelect;
export type InsertDigApiKey = typeof digApiKeys.$inferInsert;
```

Notes:

- `companyId` is the primary key ã one key per company, same relationship as the ETL enforces.
- `key_value` is stored plaintext for now. The ETL server already stores only a SHA-256 hash of the raw key, so this column is effectively a presentation cache. If Innovatr later decides to encrypt it at rest, it's a two-line change: wrap `keyValue` in an `encrypt()` / `decrypt()` in the storage helper. **Do not spend time on encryption now.**
- `onDelete: "cascade"` ã if a company is deleted, its key evaporates too.

```ts
  const [row] = await db
    .select()
    .from(digApiKeys)
    .where(eq(digApiKeys.companyId, companyId))
    .limit(1);
  if (row) return row.keyValue;

  // Fallback for local dev / bootstrapping ã only used if the table is empty for this tenant.

async setDigApiKey(companyId: string, keyValue: string): Promise<void> {
  await db
    .insert(digApiKeys)
    .values({ companyId, keyValue })
    .onConflictDoUpdate({
      target: digApiKeys.companyId,
      set: { keyValue, rotatedAt: new Date() },
    });
}

async deleteDigApiKey(companyId: string): Promise<void> {
  await db.delete(digApiKeys).where(eq(digApiKeys.companyId, companyId));
}
```

## Phase 4 ã Typed ETL client

Create `server/integrations/dig-etl.ts`. This is the only outbound-HTTP helper you'll add. Match the style of `server/emails/email-service.ts` for consistency.

```ts
const DIG_URL = process.env.DIG_ETL_API_URL ?? "https://innovatr-dig-etl.vercel.app/api/dig";

export class DigApiError extends Error {
  constructor(
    public status: number,
    public body: { code: string; message: string }
  ) {
    super(body.message);
    this.name = "DigApiError";
  }
}

async function digFetch<T>(
  path: string,
  apiKey: string,
  init?: RequestInit
): Promise<T> {
  const res = await fetch(`${DIG_URL}${path}`, {
    ...init,
    headers: {
      Authorization: `Bearer ${apiKey}`,
      "Content-Type": "application/json",
      ...(init?.headers ?? {}),
    },
  });
  if (!res.ok) {
    const body = await res.json().catch(() => ({}));
    throw new DigApiError(
      res.status,
      body.error ?? { code: "INTERNAL", message: `HTTP ${res.status}` }
    );
  }
  return res.json() as Promise<T>;
}
```

```ts
function handleDigError(res: Response, err: unknown) {

  console.error("Dig ETL error:", err);

async function resolveKey(req: AuthenticatedRequest, res: Response): Promise<string | null> {
  const companyId = req.user?.companyId;
  if (!companyId) {
    res.status(401).json({ error: { code: "UNAUTHORIZED", message: "No company context" } });
    return null;
  }
  const key = await storage.getDigApiKey(companyId);
  if (!key) {
    res.status(404).json({
      error: {
        code: "STUDY_NOT_FOUND",
        message: "No Dig ETL key configured for this company ã ask an admin to set one.",
      },
    });
    return null;
  }
  return key;
}
```

## If you get stuck

- **401 from `/api/member/dig/*`** ã the session cookie isn't valid, or `requireAuth` isn't attached to the new routes. Check the existing `/api/member/client-reports` route for reference.
- **404 from the proxy saying "No Dig ETL key configured"** ã the seed script didn't run, or it ran against the wrong database. Re-check `DATABASE_URL` and the `dig_api_keys` row.
- **401 from the ETL inside the proxy body** ã the Bearer token is wrong or revoked. Confirm with Alroy.
- **Build fails with "Module not found: @shared/schema"** ã there's already a path alias; look at how other files import from `@shared`. Don't change the tsconfig paths.
- **Recharts renders nothing** ã check that `<ResponsiveContainer>` has an explicit height and width. It will silently render empty if the parent has no layout.
- **You genuinely cannot figure something out** ã report the specific question in your final summary so the human can pick up. Don't silently work around it with a hack.
