## Architecture: server-side proxy with a single shared service token

The Innovatr Dig ETL is a separate Next.js app that ingests raw Upsiide CSV bundles for a research study and writes the parsed data into 17 tables in a sibling `dig.*` schema inside the same Neon Postgres database. You consume that data through the ETL's HTTPS API, not by reading the tables directly.

The dashboard's Express server adds 9 proxy routes under `/api/member/dig/*`. Each proxy route:

1. Runs the existing `requireAuth` middleware (reuses the dashboard's session-cookie auth — no new login needed)
2. Reads `req.user.companyId` from the authenticated session
3. Makes an HTTPS call to the corresponding `/api/dig/*` endpoint on the ETL
4. Sends two headers on every outbound call:
   - `Authorization: Service <DIG_ETL_SERVICE_TOKEN>` — a shared secret proving the caller is trusted
   - `X-Company-Id: <req.user.companyId>` — the tenant whose data to return
5. Forwards the JSON response back to the React client unchanged

The ETL's auth middleware accepts the service-token header as an alternative to the per-tenant Bearer keys it still supports for third-party consumers. When it sees a valid service token, it trusts the `X-Company-Id` header and runs every `dig.*` query with that tenant filter. **You do not need a per-tenant Bearer key. You do not need a `dig_api_keys` table. You do not need to seed anything.**

This works because the dashboard and the ETL are the same team's apps and trust each other via network + shared secret. Tenant isolation still happens — it's just asserted by the dashboard based on the authenticated session, and the ETL takes the dashboard's word for it because the service token proves the caller is legitimate.

**Three rules that don't bend:**

1. `req.user.companyId` is the only source of truth for the calling tenant. Never accept `company_id` from a request body, query parameter, or arbitrary header on your own routes.
2. The `DIG_ETL_SERVICE_TOKEN` lives only in server-side env. Never expose it to the React client, never log it, never include it in responses.
3. Read-only. The dashboard calls `GET` on 8 endpoints and `POST /search` (which is also read-only). Don't add POST/PATCH/DELETE proxy calls.

## Phase 3 — Typed HTTP client for the ETL

Create `server/integrations/dig-etl.ts`. This is the single outbound-HTTP helper. It sends both auth headers on every call.

```ts
// server/integrations/dig-etl.ts

const DIG_URL =
  process.env.DIG_ETL_API_URL ?? "https://innovatr-dig-etl.vercel.app/api/dig";

export class DigApiError extends Error {
  constructor(
    public status: number,
    public body: { code: string; message: string },
  ) {
    super(body.message);
    this.name = "DigApiError";
  }
}

function buildHeaders(companyId: string, extra?: HeadersInit): HeadersInit {
  const serviceToken = process.env.DIG_ETL_SERVICE_TOKEN;
  if (!serviceToken) {
    throw new Error(
      "DIG_ETL_SERVICE_TOKEN is not set — add it to Replit Secrets",
    );
  }
  return {
    Authorization: `Service ${serviceToken}`,
    "X-Company-Id": companyId,
    "Content-Type": "application/json",
    ...(extra ?? {}),
  };
}

async function digFetch<T>(
  path: string,
  companyId: string,
  init?: RequestInit,
): Promise<T> {
  const res = await fetch(`${DIG_URL}${path}`, {
    ...init,
    headers: buildHeaders(companyId, init?.headers),
  });
  if (!res.ok) {
    const body = await res.json().catch(() => ({}));
    throw new DigApiError(
      res.status,
      body.error ?? { code: "INTERNAL", message: `HTTP ${res.status}` },
    );
  }
  return res.json() as Promise<T>;
}

// ——— Response type aliases (server side copies of the client types)
// Keep them separate from client/src/lib/dig-api.types.ts so the server
// doesn't import from the client tree. Same shapes, different home.

export interface DigStudy {
  id: string;
  title: string;
  source_study_name: string | null;
  public_client_report_id: string | null;
  ingest_status: string;
  file_count: number;
  respondent_count: number;
  concept_count: number;
  created_at: string;
  updated_at: string;
}

export interface DigSearchRequest {
  query: string;
  study_id?: string;
  concept_id?: string;
  limit?: number;
}

// All response envelopes are `{ key: data }` — keep them typed loosely here
// because the client has the strict types. The server's job is to pass through.

export const digApi = {
  listStudies: (companyId: string) =>
    digFetch<{ studies: DigStudy[] }>("/studies", companyId),

  getStudy: (companyId: string, id: string) =>
    digFetch<{ study: unknown }>(`/studies/${id}`, companyId),

  listConcepts: (companyId: string, studyId: string) =>
    digFetch<{ concepts: unknown[] }>(`/studies/${studyId}/concepts`, companyId),

  getConcept: (companyId: string, studyId: string, conceptId: string) =>
    digFetch<{ concept: unknown }>(
      `/studies/${studyId}/concepts/${conceptId}`,
      companyId,
    ),

  getHeatmap: (companyId: string, studyId: string, conceptId: string) =>
    digFetch<{ clicks: unknown[] }>(
      `/studies/${studyId}/concepts/${conceptId}/heatmap`,
      companyId,
    ),

  getRanking: (companyId: string, studyId: string) =>
    digFetch<{ ranking: unknown[] }>(`/studies/${studyId}/ranking`, companyId),

  getDemographics: (companyId: string, studyId: string) =>
    digFetch<{ demographics: unknown }>(
      `/studies/${studyId}/demographics`,
      companyId,
    ),

  getThemes: (companyId: string, studyId: string) =>
    digFetch<{ themes: unknown[] }>(`/studies/${studyId}/themes`, companyId),

  search: (companyId: string, body: DigSearchRequest) =>
    digFetch<{ results: unknown[] }>("/search", companyId, {
      method: "POST",
      body: JSON.stringify(body),
    }),
};
```

The `unknown[]` / `unknown` typing on the server is intentional — the server is a pass-through. The client's types in `dig-api.types.ts` carry the contract all the way to the components.

```ts
        credentials: "include",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(body),
      });
      if (!res.ok) throw new Error(await res.text());
      return res.json();
    },
  });
}
```

The trick: `queryKey: ["/api/member/dig/studies", studyId, "concepts"]` joins via the default fetcher in `client/src/lib/queryClient.ts` to `/api/member/dig/studies/<uuid>/concepts`. This is exactly how the existing `/api/member/*` queries work.

```tsx
    existing.neutral += t.neutral;
    existing.negative += t.negative;
```

## Style + constraints

- **Match the repo's existing style.** Look at neighbouring files before writing any new code.
- **TypeScript strict.** No new `any` except where the server pass-through is deliberately loose.
- **Error UI is not optional.** Every component handles loading + error + empty states.
- **No new dependencies.** Everything's already installed.
- **Don't touch unrelated code.** Surgical edits.
- **Service token stays on the server.** Never exposed to the client.
- **Read-only.** All 9 routes are GET (or POST for search, which is still read-only). No mutations against the ETL.
- **One commit per phase.** Conventional commits. e.g. `feat(dig): add typed HTTP client for ETL API`, `feat(dig): add 9 proxy routes under /api/member/dig/*`.

## What NOT to do

- **Do not add a `dig_api_keys` table** to `shared/schema.ts`. Zero schema changes.
- **Do not add per-tenant Bearer keys.** One shared service token is the whole auth story.
- **Do not read `dig.*` tables directly from Drizzle.** The dashboard goes through the HTTP API on every call.
- **Do not expose `DIG_ETL_SERVICE_TOKEN` to the React client.** Never as a prop, never in a response body, never in a log line.
- **Do not add a new charting library.** Recharts is already installed.
- **Do not skip `requireAuth` on any proxy route.** Every single one is session-gated.
- **Do not change the JSON response envelope shape** (`{ studies: [...] }`, `{ concept: {...} }`, `{ error: { code, message } }`).

## If you get stuck

- **401 on the dashboard proxy** — `requireAuth` didn't fire or the session is expired. Compare against `/api/member/client-reports`.
- **401 `{"error":{"code":"UNAUTHORIZED"}}` from the ETL in the response body** — the service token is wrong or missing. Check Replit Secrets.
- **Error "DIG_ETL_SERVICE_TOKEN is not set"** — the env var didn't reach the Express process. Restart the dev server after adding.
- **500 with a fetch/network error** — the dashboard can't reach the ETL URL. Check `DIG_ETL_API_URL` (if overridden) or test with `curl https://innovatr-dig-etl.vercel.app/api/health` from the Replit shell.
- **Empty `{studies: []}`** — `req.user.companyId` doesn't match a tenant that has ingested data on the ETL. Confirm the session user is in the same company.
- **Recharts renders nothing** — `<ResponsiveContainer>` needs an explicit height. Check the parent.
- **You genuinely cannot figure something out** — report the specific question in your final summary. Don't silently work around it.
