1. Context and Goals

2. Data Model and Storage Requirements
2.1 Credits
Confirm or create a persistent data model that stores credits at company level (or user level if that is what already exists). For example:
Company {
  id: string;
  name: string;
  // Existing fields...
  test24BasicCredits: number;
  test24ProCredits: number;
}

Or, if you already use a separate Credits table, reuse that and extend it if needed.
Requirements:


Credits must be persisted in the database, not in memory.


Credits must be decremented atomically when used to pay for a project, to prevent race conditions if two tabs or two admins trigger actions at the same time.


If there is not enough credit of the correct type, the API must reject the request with a clear error and leave the balances unchanged.



2.2 Research Projects / Briefs
Ensure each submitted brief creates or updates a persistent record, for example:
ResearchProject {
  id: string;
  companyId: string;
  studyType: 'Test24 Basic' | 'Test24 Pro';
  numberOfConcepts: number;
  paymentMethod: 'pay_online' | 'invoice' | 'credits';
  basicCreditsUsed: number; // normally 0 or numberOfConcepts
  proCreditsUsed: number;   // normally 0 or numberOfConcepts
  status: 'new' | 'audience_live' | 'analysing_data' | 'completed';
  createdAt: Date;
  updatedAt: Date;
  // Other existing fields like title, tags, links to reports, etc
}

If a similar model already exists, extend it instead of creating a new one. Persist the chosen payment method and the number of credits consumed (if any) for auditability.

4. Credit Deduction Logic
When Use My Credits is the selected payment method and the user submits the brief:


The front end should send paymentMethod: 'credits' in the payload plus any necessary metadata (studyType, companyId, numberOfConcepts, etc).


The API must:


Start a transaction.


Lock or safely update the company row or credits row to avoid concurrent modification issues.


Compute the credits required, normally:


If studyType = 'Test24 Basic' then basicCreditsRequired = numberOfConcepts and proCreditsRequired = 0.


If studyType = 'Test24 Pro' then proCreditsRequired = numberOfConcepts and basicCreditsRequired = 0.




Check balances.


If insufficient, abort and return a clear error message (see above).


If sufficient:


Subtract required credits from the company balance.


Create a ResearchProject (or equivalent) record with paymentMethod = 'credits' and basicCreditsUsed / proCreditsUsed fields set.




Commit the transaction.




Return success to the front end.


On success, the client should:


Show the existing “brief submitted successfully” confirmation, plus:
We’ve deducted your credits and moved your project into the queue. You’ll receive an email shortly confirming everything. You can track your project anytime in the My Research tab.


Important: credits must only be deducted once per accepted submission. Refreshing the page must not cause double deduction.

7. Performance and Optimisation


Add appropriate indexes on key fields: companyId, status, createdAt for fast filters in My Research.


Paginate results in My Research and any admin view to avoid loading huge lists in one go.


When calculating credit balances, query only the necessary data (do not hydrate large unrelated objects).


Keep all heavy logic for credit checks and deductions on the server to avoid exposing sensitive data or business rules in the client.



8. Error Handling and Edge Cases


If the credit validation API fails for any reason, return a user friendly message and prevent credit deduction. The UI should show a message like:
Something went wrong while validating your credits. Please try again or select another payment method. If the problem persists, contact Hannah at hannah@innovatr.co.za.


If email sending fails but the brief and credits have been stored successfully:


Do not roll back the brief.


Log the error and surface it in admin tooling if possible.


Admins can still see the project in My Research and follow up manually.




Ensure that resubmitting or refreshing after a successful submission does not create duplicate projects or double deduct credits. This may require idempotency at either front end or API level.
